8. OBSERVACIONES Y NOTAS TÉCNICAS
Configuración especial requerida Java 21 + Lombok 1.18.38: Se requirió configurar annotationProcessorPaths en el pom.xml y crear lombok.config por compatibilidad con Java 21.
Mockito subclass: Se requirió src/test/resources/mockito-extensions/org.mockito.plugins.MockMaker con valor mock-maker-subclass para evitar restricciones de byte-buddy en Java 21.
reactor-test: Se usó StepVerifier para testear los Mono<ResponseEntity> del orquestador WebFlux reactivo.
pytest-cov: Comando: pytest --cov=app --cov-report=html:htmlcov tests/
Estrategia de mocking Todos los tests unitarios utilizan mocks de Mockito para aislar las dependencias externas (repositorios JPA, MongoDB, Kafka producers, MinIO). Los servicios de dominio se instancian directamente. No se usan bases de datos ni brokers reales en las pruebas unitarias.
Cobertura de seguridad Se validaron controles de acceso en micro-mascotas (MSC-013, MSC-016): un usuario no propietario que intenta modificar o eliminar el reporte de otro usuario recibe SecurityException. En orquestador se validan 5 escenarios de JWT (ORQ-001 a ORQ-008) incluyendo tokens expirados, malformados y con firma incorrecta.
Ambiente de despliegue paralelo Paralelamente a las pruebas unitarias, la aplicación fue desplegada en Kubernetes (Docker Desktop k8s v1.34.1) con 11 pods activos en namespace sanos-salvos. API disponible en http://localhost:30080, frontend en http://localhost:4321.
Siguiente fase Pruebas de integración con Testcontainers (PostgreSQL + Kafka en Docker) y pruebas E2E con Playwright para el frontend Astro/React.
